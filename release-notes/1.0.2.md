# unicode_ranges v1.0.2

`unicode_ranges` `v1.0.2` is a patch release that expands the view/match surface, fixes owning-string predicate replacement overload visibility, and adds more runtime-path optimizations while preserving the `1.x` source contract.

## Changed

- Added the forward `match_indices(...)` member family for UTF-8, UTF-16, and UTF-32 strings and string views, completing the counterpart to `rmatch_indices(...)`. The match family now also accepts copyable forward ranges of character values as lazy "match any of these characters" delimiter sets, with temporary arrays owned by the returned view.
- Fixed owning-string predicate replacement overload visibility so `replace_all(pred, to)` and `replace_n(count, pred, to)` work directly on UTF-8, UTF-16, and UTF-32 owning strings.
- Optimized owning-string insert paths to avoid temporary transcoding buffers for repeated character inserts and cross-encoding `insert_range(...)` overloads. UTF-8, UTF-16, and UTF-32 owning strings now pre-measure the inserted range, grow the destination once, move the existing tail once, and write directly into the insertion gap where the source encoding cannot alias the destination buffer.
- Rvalue owning strings now return move-only owning views from `chars()`, `reversed_chars()`, `char_indices()`, `grapheme_indices()`, `graphemes()`, the split-family view methods, and the match-family view methods, avoiding dangling views when a view is kept from a temporary owning string. String-view overloads remain borrowed views.
- Optimized owning-string materialization from same-encoding `chars()` and rvalue `reversed_chars()` views: `std::from_range`, `append_range`, `assign_range`, `insert_range`, and `replace_with_range_inplace` now use direct same-encoding storage paths and can reuse storage from rvalue owning views when allocator compatibility allows it.
- Expanded the compiled simdutf runtime bridge for UTF-16/UTF-32 validation, UTF-16/UTF-32 bulk conversion paths, cross-encoding range mutation/materialization, guarded ASCII-only checks, and UTF-8/UTF-16 character counting while preserving scalar constexpr fallbacks.
